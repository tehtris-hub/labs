Suricata —
IDS Réseau & Règles

Suricata est installé sur la VM mais ne génère aucune alerte. L'interface réseau n'est pas configurée et les règles Emerging Threats sont désactivées. Votre mission : corriger la configuration YAML et activer les règles communautaires.

01 //

Objectif

Suricata est un IDS/IPS réseau open-source. Il inspecte le trafic en temps réel et génère des alertes basées sur des règles (fichiers .rules).

À la fin de ce lab vous saurez :

  • Identifier l'interface réseau de votre VM
  • Configurer suricata.yaml correctement
  • Télécharger les règles Emerging Threats avec suricata-update
  • Tester Suricata en mode IDS et lire les alertes
02 //

État de la VM — Ce qui ne fonctionne pas

Problèmes détectés

(1) Dans /etc/suricata/suricata.yaml, l'interface réseau est réglée sur eth99 qui n'existe pas. (2) La section rule-files ne pointe vers aucun fichier. (3) suricata-update n'a jamais été exécuté — aucune règle n'est présente.

Diagnostiquez d'abord :

Diagnostic initial
# Trouver l'interface réseau active de la VM
ip link show
ip addr

# Voir l'interface configurée dans suricata.yaml (ligne af-packet)
grep -n "interface" /etc/suricata/suricata.yaml | head -5

# Vérifier les règles présentes
ls /var/lib/suricata/rules/

# Tenter de démarrer et observer l'erreur
sudo suricata -T -c /etc/suricata/suricata.yaml
À NOTER La commande suricata -T teste la configuration sans démarrer le moteur. C'est votre meilleur outil de diagnostic : elle affiche exactement ce qui est mal configuré.

04 //

Vérification — Générer une Alerte de Test

Lancez Suricata en mode IDS sur votre interface et déclenchez une alerte connue :

Vérification
# Démarrer Suricata en mode IDS (remplacer eth0 par votre interface)
sudo suricata -c /etc/suricata/suricata.yaml -i eth0 -D

# Générer du trafic de test (scan nmap local)
nmap -sS 127.0.0.1

# Lire les alertes en temps réel
sudo tail -f /var/log/suricata/fast.log

# Ou lire les logs JSON structurés (plus complets)
sudo tail -f /var/log/suricata/eve.json | python3 -m json.tool | grep -A5 '"event_type":"alert"'
SUCCÈS ATTENDU Le fichier fast.log doit contenir des lignes d'alerte avec un timestamp, la priorité, et la signature déclenchée. Si fast.log se remplit, Suricata inspecte bien le trafic.
